Protokolle
IP
IP steht für Internet Protocol. Es managed die gesamte logische Adressierung im Internet, sodass jeder Rechner, Drucker, jedes Handy oder anderwertiges Gerät eine IP-Adresse bekommt. Dabei arbeitet IP auf der Schicht 3 im OSI-Schichtenmodell, der Vermittlungsschicht.

Man unterscheidet bislang zwei Arten von IP: IPv4 und IPv6.
IPv4
IPv4 ist Version 4 von IP. Es wird noch in großen Teilen Europas, der USA und vielen anderen Teilen der Welt verwendet, obwohl es eigentlich nicht mehr genug IPv4-Adressen zur Verfügung hat. Das ist deshalb so, weil zur Zeit der Erfindung von IPv4 noch niemand dachte, dass jeder Mensch einmal ca. 3 technische Geräte haben wird und all diese Geräte einzigartige IP-Adressen brauchen, um untereinander kommunizieren zu können.

Aufbau
Eine IPv4-Adresse ist folgendermaßen aufgebaut:
Schreibweise Beispiel einer IPv4-Adresse
Binär 11000000 10101000 10001001 00001011
Decimal 192 168 137 11
Hexadecimal C0 A8 89 0B
Wie Sie sehen können besteht jede IPv4-Adresse insgesamt aus 32 Bit / 4 Byte und nur aus Einsen und Nullen. Diese Schreibweise eignet sich besonders gut für Computer, weil 1 für "Strom fließt" und 0 für "kein Strom fließt" stehen kann. Jedoch sind Menschen eher mit dem Decimalsystem vertraut. Aus diesem Grund kann man 8 Bit / 1 Byte (binär) einfach in eine Zahl zwischen 0 und 255 (decimal) umrechnen.

Diese 32 Bit kann man in den Netzanteil (ersten 16 Bit / 2 Byte) und Hostanteil (hinteren 16 Bit / 2 Byte) unterteilen.

normales ARP
Was ist das ARP?
Das Address Resolution Protocol wandelt IP-Adressen in MAC-Adressen um, sodass Pakete von Rechner zu Rechner geschickt werden können.

Die IP-Adressen sind für die logische Adressierung zuständig während die MAC-Adressen für die physische Adressierung zuständig sind. Aus diesem Grund werden alle Pakete in Wahrheit immer mit MAC-Adressen von Netzwerk-Adapter zu Netzwerk-Adapter gesendet.
OSI-Schichtenmodell
Das Address Resolution Protocol arbeitet auf Layer 2 im OSI-Schichtenmodell. Manchmal wird behauptet, ARP wäre ein Schicht-3-Protokoll, allerdings ist Adressauflösung eine Aufgabe der Schicht 2. Ganz korrekt wäre eine Zuordnung zwischen Schicht 2 und 3.
ARP - Schicht
Layer 2: Sicherungsschicht
(Layer 3: Vermittlungs- / Paketschicht)
Funktionsweise
Das ARP sendet zum Herausfinden der MAC-Adresse des Ziel-Computers ein ARP-Request an die MAC-Broadcast-Adresse "FF-FF-FF-FF-FF-FF".

In diesem Paket ist die IP-Ziel-Adresse des eigentlichen Paketes enthalten. Fühlt sich ein Rechner mit dieser IP-Adresse angesprochen, so sendet er ein ARP-Reply Paket an die MAC-Quell-Adresse zurück. Die MAC-Adresse dieses Rechners speichert der ursprüngliche Rechner zur schnelleren ARP-Adressauflösung im ARP-Cache.
RARP
RARP steht für Reverse Address Resolution Protocol. Es ist eine Variante von ARP, die ganz einfach MAC-Adressen in IP-Adressen umwandelt.

ICMP
Was ist das ICMP?
Das Internet Control Message Protocol ist ein wesentlicher Bestandteil des Internets. Es wird allerdings nicht als Teil des IP-Protokolls angesehen, sondern als eigenes Protokoll, das zur Übermittlung von Meldungen über IP dient.

ICMP regelt vor allem Fehlermeldungen und Statusinformationen der Protokolle TCP, UDP und IP. Die ICMP-Meldungen werden zwischen Rechnern und aktiven Netzknoten, zum Beispeil Routern, benutzt, um sich gegenseitig Probleme mit Datenpaketen mitzuteilen. Ziel ist, die Übertragunsqualität zu verbessern.
OSI-Schichtenmodell
Das Internet Control Message Protocol arbeitet auf Layer 3 im OSI-Schichtenmodell.
ICMP - Schicht
Layer 3: Vermittlungs- / Paketschicht
Anwendungen
Als Benutzer bekommt man von ICMP eigentlich nichts mit. Jedoch hat jedes TCP/IP-Netzwerk Tools, wie zum Beispiel Trace oder Ping, die ICMP verwenden.
Trace
Das Trace-Tool sendet von jedem Rechner, Router oder Switch ein ICMP-Paket zurück, bis es beim Ziel angekommen ist. Damit soll man Fehler schneller finden können. Wenn das Paket bei über 30 Routern vorbei muss, wird es gelöscht, weil die TTL-Zeit (Time To Live) bei jedem Router erhöht wird, bis sie bei 30 angekommen ist.
Firewall
Theorie
Was ist die Firewall?
Die Firewall filtert Anfragen bei Routern und Rechnern, sodass nicht alle Pakete durchkommen, sondern nur die Pakete, die erlaubt sind.
Funktionen der Firewall
Stateful Inspection
Stateful Inspection dient allgemein zur Beschleunigung des Verbindungsaufbaus. Wenn eine IP-Adresse schon einmal verbunden war, sorgt Stateful Inspection dafür, dass die IP-Adresse im Cache gespeichert wird, um zukünftige Verbindungen schneller aufbauen zu können.
Connection Tracking
Connection Tracking bedeutet auf Deutsch Verbindungsverfolgung. Diese Funktion ist essenziell, damit zum Beispiel Webserver funktionieren. Connection Tracking sorgt nach dem Aufbau einer Verbindung mit einem Server dafür, dass die IP-Adressen auf einen anderen Port umgeleitet werden. Diese Verlagerung des Portes gibt den ursprünglichen reservierten Port wieder für andere Clients frei.
NAT Masquerade
Eine weitere Funktion der Firewall ist Masquerade. Also IP-Adressen umwandeln beziehungsweise austauschen. Zum Beispiel tauscht der Router die Ziel-Adresse (Adresse des Routers) durch die Adresse des Rechners aus, welchen er mittels Portnummer herausfindet.
Übersicht / Ablauf
Im folgenden Bild sehen Sie den allgemeinen Ablauf der Firewall. Jedes Paket muss einen dieser Pfade durchqueren. Die iptables-Architektur gruppiert die Regeln in vier Tabellen. Diese Tabellen enthalten Chains (von Verarbeitungsregeln), bestehend aus Patterns (welche Pakete) und Targets (was mit dem Paket passieren soll). Anders gesagt bestimmen also Chains wo geprüft wird, Patterns welche Pakete betroffen sind und Targets was mit ihnen geschieht.
Tabellen / Tables
Die Firewall managed vier verschiedene Tabellen:
Tabelle Einsatzgebiet
Filter Dies ist die Standardtabelle. Wie der Name vermuten lässt, werden hier alle "reinen" Filterregeln hinterlegt.
NAT Diese Tabelle wird in der Regel für Adressumsetzung (NAT = Network Address Translation) und spezielle Weiterleitungen (Port Forwarding) eingesetzt.
Mangle Diese Tabelle wird in der Regel bei gewünschter Paketmanipulation eingesetzt.
Raw Diese Tabelle wird in der Regel eingesetzt, um Ausnahmen vom Connection Tracking zu definieren.
Ketten / Chains
In der Firewall gibt es fünf verschiedene sogenannte Chains.
Chain Wann diese Chain an der Reihe ist für Tabellen Beispiele
Prerouting Diese Chain wird bereits vor dem Routen aufgerufen. raw, mangle, nat Portforwarding (= Ziel-Adresse auf internes Netz ändern)
Forward Diesen Schritt durchlaufen alle Pakete, die geroutet werden. mangle, filter nur Administrator kann geroutet werden und somit ins Internet
Input Hier landen alle Pakete, die an einem lokalem Prozess gerichtet sind. mangle, nat, filter nur Pakete auf Port 80
Output Hier laufen alle Pakete durch, die von einem lokalem Prozess stammen. raw, mangle, nat, filter nur Pakete von meinem Rechner
Postrouting Diesen Schritt passieren alle Pakete, entweder nach Output oder nach Forward. mangle, nat Masquerade (= IP-Adressen umwandeln)
Ziele / Targets
Jedes Paket besteht aus einer Spezifikation und einem Ziel. Das Ziel gibt letztendlich an, was mit dem Paket passieren soll, zum Beispiel akzeptieren und weiterleiten oder fallen lassen und wegwerfen.

Das ist eine Tabelle mit den wichtigsten Zielen:
Ziel Beschreibung gültig
ACCEPT Das Paket wird durchgelassen.
DROP Das Paket wird ohne Rückmeldung verworfen.
QUEUE Das Paket wird an eine Queue im Userspace umgeleitet. Dort kann es bearbeitet werden.
RETURN Leitet das Paket an die nächste Chain weiter.
LOG Informationen werden in Syslog gespeicher. Das Paket wird weitergeleitet.
REJECT Das Paket wird mit Rückmeldung verworfen (Rückmeldung mittels ICMP).
DNAT Destination NAT ändert die Ziel-Adresse des Pakets. NAT
SNAT Source NAT ändert die Quell-Adresse des Pakets. NAT
MASQUERADE Mischung: DNAT / SNAT (je nach dem) NAT
REDIRECT Leitet das Paket zum lokalen Rechner um. NAT
Muster / Pattern
Das Muster enthält die Bedingung, die das Paket erfüllen muss, um das Ziel zu erreichen.
Muster (Commands) Beschreibung
-A, --append chain rule-specification Regel wird an die Kette "Chain" angehängt.
-C, --check chain rule-specification Prüft, ob es die Chain bereits gibt.
-D, --delete chain rule-specification

-D, --delete chain rulenum 		Regel wird aus der Kette "Chain" gelöscht.
-I, --insert chain [rulenum] rule-specification Regel wird an Position "Position" der Kette "Chain" eingefügt. Bei Nichtangabe der Position wird die Regel am Anfang der Kette eingefügt.
-R, --replace chain rulenum rule-specification Die Regeln in der angegebenen Chain wird ersetzt.
-L, --list [chain] Liste alle Regeln der Kette "Chain" auf.
-F, --flush [chain] Alle Regeln der Kette "Chain" löschen.
-N, --new-chain chain Neue Chain erstellen
-X, --delete-chain [chain] Chain löschen.
-P, --policy chain target Policy für die Chain setzen.
-E, --rename-chain old-chain new-chain Chain umbenennen.
Muster (Parameters) Beschreibung
-p, --protocol protocol Das Paket wird nur durchgelassen, wenn es gemäß "IP-Protokoll" ist (z.B. TCP, UDP, ICMP, Spezialwort: "all").
-s, --source address[/mask][,...] Das Paket wird nur durchgelassen, wenn es von der definierten IP-Adresse stammt.
-d, --source address[/mask][,...] Das Paket wird nur durchgelassen, wenn es an die definierte IP-Adresse gesendet wird.
-m, --match match Prüft das Paket auf erweiterte Spezifikationen / Eigenschaften.
-j, --jump target Prüft das Ziel der Regel auf erweiterte Spezifikationen / Eigenschaften.
-g, --goto chain Der Prozess soll in einer bestimmten Chain fortsetzen.
-t table Diese Filterregel gilt für die Tabelle "table".
-i, --in-interface name Das Paket wird nur durchgelassen, wenn es über die definierte Netzwerkschnittstelle eingegangen ist.
-o, --out-interface name Das Paket wird nur durchgelassen, wenn es über die definierte Netzwerkschnittstelle versendet wird.
--sport, --source-port <Port-Nr> Das Paket wird nur durchgelassen, wenn es von der definierten Port-Nummer stammt. Muss zwingend in Verbindung mit -p benutzt werden!
--dport, --destination-port <Port-Nr> Das Paket wird nur durchgelassen, wenn es an die definierte Port-Nummer gesendet wird. Muss zwingend in Verbindung mit -p benutzt werden!
Frontends / Alternativen
Damit die iptables-Befehle nicht alle per Hand in die Console geschrieben werden müssen, gibt es heutzutage bereits Frontends beziehungsweise Alternativen. Zum Beispiel gibt es grafische Oberflächen, die im Hintergrund die iptables-Befehle ausführen.
Praxis
Beispiel
Ausgangssituation
Sie wollen eine Firewall konfigurieren. Grundsätzlich soll kein Paket durchgelassen werden bis auf folgende:
loopback, http, https, proxy, smtp, imap, pop3, masquerade, Remoteverbindungen, Admin
Konfiguration löschen
Damit sich die Konfiguration nicht mit einer bereits vorhandenen Konfiguration überschreibt, löschen wir zuerst die gesamte Konfiguration.

Bitte beachten Sie, dass diese Befehle ganz am Anfang und keinesfalls am Ende ausgeführt werden sollen!
# alles löschen
iptables -t filter -F
iptables -t nat -F
Grundsätzliche Einstellungen
Anschließend wollen Sie einstellen, dass alle Pakete, die keine spezifische Konfiguration bekommen, gedropt werden sollen. Und die Firewall soll Conection Tracking erlauben (Wenn die Portnummer geändert wird, soll die Verbindung aufrecht erhalten bleiben).
# default policy
iptables -t filter -P INPUT DROP
iptables -t filter -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

iptables -t filter -P FORWARD DROP
iptables -t filter -A FORWARD -m state --state ESTABLISHED,RELATED -j ACCEPT
Loopback
Diese Einstellung ist notwendig, damit die Firewall Paket von loopback animmt.
# loopback
iptables -t filter -A INPUT -i lo -j ACCEPT
http, https
Damit http und https Paket angenommen werden, müssen wir auch diese Konfigurationen vornehmen.
Dies ist zum Beispiel bei einem Webserver wichtig.
# http, https
iptables -t filter -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -t filter -A INPUT -p tcp --dport 443 -j ACCEPT
Proxy
# proxy
iptables -t filter -A INPUT -p tcp --dport 3128 -s 10.10.10.0/24 -j ACCEPT
Mail
Die folgenden Konfigurationen sind für smtp, imap und pop3 zuständig.
# smtp, smtps
iptables -t filter -A INPUT -p tcp --dport 25 -s 10.10.10.0/24 -j ACCEPT
iptables -t filter -A INPUT -p tcp --dport 465 -j ACCEPT
iptables -t filter -A INPUT -p tcp --dport 587 -j ACCEPT
# imap, imaps
iptables -t filter -A INPUT -p tcp --dport 143 -s 10.10.10.0/24 -j ACCEPT
iptables -t filter -A INPUT -p tcp --dport 993 -j ACCEPT
# pop3, pop3s
iptables -t filter -A INPUT -p tcp --dport 110 -s 10.10.10.0/24 -j ACCEPT
iptables -t filter -A INPUT -p tcp --dport 995 -j ACCEPT
NAT
Damit im Falle der Verwendung von Masquerade alle Pakete weitergeleitet können.
# masquerade (NAT)
iptables -t nat -A POSTROUTING -o ens33 -j MASQUERADE
Admin
Damit alle Pakete von Admin raus geroutet werden können.
# forwarding
iptables -t filter -A FORWARD -s 10.10.10.50 -j ACCEPT
Remoteverbindung
Damit Mitarbeiter von zu Hause aus arbeiten können (Remotedesktopverbindung).
# port-forwarding
iptables -t nat -A PREROUTING -p tcp --dport 3389 -j DNAT --to-destination 10.10.10.50:3389

Anwendung auf Server
Es gibt mehrere verschiedene Möglichkeiten, wie man diese Konfigurationen nun auf der Firewall dauerhaft speichert. Im folgenden werden zwei wichtige bekannte Methoden erklärt:
Methode 1: Shell-Script
Bei der ersten Methode schreibt man sich ein Script, welches jedes Mal beim hochfahren des Rechners automatisch startet. Diese Methode funktioniert mithilfe SystemV (= Manager, der Shellscripte oder Anwendungen startet).
TXT-Datei erstellen
Hierfür erstellen Sie eine einfache eine neue leere Datei mit .sh-Endung oder gar keiner Endung, wo Sie zum Beispiel die obrige Konfigurationen hineinkopieren. Achten Sie bitte darauf, dass Sie Netzwerkkarten und IP-Adressen noch ändern müssen, wenn Sie die obrige Konfiguration verwenden wollen.
Link erstellen
Anschließend erstellen Sie in /etc/init.d einen Symbollink auf die Datei:
ln -s /etc/init.d/startscript /etc/rc2.d/S01startscript
Bei der Namensgebung des Links ist folgendes zu beachten: S steht für Start, K für Kill. Die Nummer hinter dem Buchstaben ist nur wichtig, wenn Prozesse voneinander abhängig sind und ein Prozess erst später ausgeführt werden kann, wenn ein anderer Prozess erst fertig ist.
Methode 2: InitD-Service
Eine andere Möglichkeit die Firewall dauerhaft zu konfigurieren, jedoch kein Script verwenden zu müssen (weil das nicht jeder mag), ist mithilfe eines SystemD-Services.

Hierfür muss man zuerst alle iptables-Befehle in der Console ausführen.
Speichern
Damit diese Konfiguration nun auch gespeichert wird, führen Sie diesen Befehl aus (Hinweis: Speichern Sie die Konfiguration im Ordner /etc/iptables):
iptables-save -f iptables.rules
Laden
Wenn Sie nun die Konfigurationen laden wollen, werden Sie diesen Befehl:
iptables-restore iptables.rules
Automatisches Laden beim Hochfahren
Damit Sie nicht jedes Mal nach dem Hochfahren die Konfiguration per Hand laden müssen, können Sie auch einstellen, dass diese automatisch geladen wird, und die Firewall somit automatisch konfiguriert ist. Dies wird mithilfe eines „systemd“-Service umgesetzt.

Als erstes erstellen Sie eine empty.rules Datei, deren Inhalt leer ist. Diese Datei wird verwendet, wenn der iptables-Service deaktiviert ist (vergleichbar mit "iptables -F").

Erstellen Sie eine neue Datei: „/etc/systemd/system/iptables.service“. Kopieren Sie folgenden Inhalt in diese Datei: Achten Sie wieder auf die Dateipfad und -namen!
[Unit]
Description=iptables firewall service
After=network.target

[Service]
Type=oneshot
ExecStart=/sbin/iptables-restore /etc/iptables/iptables.rules
RemainAfterExit=true
ExecStop=/sbin/iptables-restore /etc/iptables/empty.rules

[Install]
WantedBy=multi-user.target
Schlussendlich führen Sie noch diesen Befehl aus, damit die Datei beim Hochfahren ausgeführt wird:
systemctl enable --now iptables.service
